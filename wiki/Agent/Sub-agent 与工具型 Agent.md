# Sub-agent 与工具型 Agent

## 概述

在多 Agent 系统中，sub-agent 与“agent as tool”的差异不在于是否使用 LLM，而在于控制权和上下文边界：工具型 Agent 适合离散、无状态、可复用任务；sub-agent 适合复杂、有状态、依赖共享上下文的多步骤子问题。

## 核心内容

### 两种模式的本质差异

**工具型 Agent** 像一个可调用的专业函数：主 Agent 给出明确输入，工具在隔离会话中完成任务并返回结果。调用方不关心内部推理过程，只依赖稳定的输入输出契约。

**Sub-agent** 像一个被委派的团队成员：父 Agent 交给它一个较高层级的子目标，它在父任务的上下文中进行多步骤推理、交互和状态维护。

核心分界线：

| 维度 | 工具型 Agent | Sub-agent |
| --- | --- | --- |
| 任务复杂度 | 低到中，原子能力 | 高，多步骤工作流 |
| 上下文 | 隔离，无会话历史 | 共享或继承父任务上下文 |
| 状态 | 无状态，请求内自足 | 有状态，跨步骤累积决策 |
| 复用性 | 高，可被多个 Agent 调用 | 中低，常绑定特定流程 |
| 控制权 | 主 Agent 控制流程 | 父 Agent 委派子问题 |
| 接口形态 | 严格输入输出 | 目标导向、过程自治 |

### 使用工具型 Agent 的场景

当任务满足以下特征时，把 Agent 包装成工具更合适：

- 输入和输出可以被清晰定义。
- 单次调用即可完成，不需要父会话历史。
- 能力具有通用复用价值。
- 主 Agent 需要保留流程编排权。
- 失败模式可以通过工具契约和重试策略处理。

例子：

- NL2SQL：输入自然语言问题、数据库 schema，输出 SQL。
- 数据库执行器：输入 SQL，输出结构化结果。
- 文档分类器：输入文档，输出类别和置信度。
- 格式转换器：输入数据，输出指定格式。

工具型 Agent 的风险是：如果任务实际需要大量上下文，每次调用都要把完整背景塞进输入，容易造成 token 冗余、状态丢失和接口膨胀。

### 使用 Sub-agent 的场景

当任务满足以下特征时，使用 sub-agent 更合适：

- 目标本身较模糊，需要连续推理和澄清。
- 任务内部包含多步操作和中间决策。
- 子任务必须理解父任务目标、历史偏好或当前约束。
- 需要一定自治，而不是简单请求-响应。
- 结果质量依赖上下文连续性。

例子：

- 数据可视化 Agent：根据用户问题和查询结果选择图表类型、生成代码、执行和修正图表。
- 旅行规划中的航班、酒店、行程 Agent：需要持续跟踪日期、预算、偏好和已确认决策。
- 复杂编码任务中的探索、诊断、审查 Agent：需要围绕项目目标连续阅读、推理、验证。

Sub-agent 的风险是耦合更高、复用性更低，并且父子 Agent 之间的上下文边界、权限和完成条件必须设计清楚。

### 决策准则

可以用一个问题快速判断：**如果只给它一个 JSON 输入，能否可靠返回一个 JSON 输出？**

- 可以：优先做成工具型 Agent。
- 不可以，因为它需要对话历史、跨步状态或自主探索：优先做成 sub-agent。

更细的判断：

1. **任务是否原子？** 原子任务用工具；复合目标用 sub-agent。
2. **是否需要共享上下文？** 不需要则工具；需要则 sub-agent。
3. **是否要复用？** 通用能力做工具；流程内角色做 sub-agent。
4. **谁控制流程？** 主 Agent 控制则工具；子 Agent 自治推进则 sub-agent。
5. **状态在哪里？** 若状态可由调用方显式传入，工具足够；若状态需要长期维护，sub-agent 更自然。

### 与 Harness Engineering 的关系

在 [[Harness Engineering]] 中，工具和 sub-agent 都是 Harness 的组成部分，但承担不同约束：

- 工具提供稳定、可测试、可复用的能力边界。
- Sub-agent 提供复杂任务的分工、上下文隔离和认知减负。

好的 Harness 不应把所有能力都做成 sub-agent。过度委派会增加通信和监督成本；过度工具化则会让主 Agent 背负过多上下文搬运和流程管理负担。
