## 核心原则

1. **范式化设计（Normalization）** — 通过 1NF（原子性）、2NF（消除部分依赖）、3NF（消除传递依赖）减少数据冗余，预防插入/更新/删除异常。
2. **正确使用主键和外键（Keys & Constraints）** — 每个表必须有主键，外键关系必须定义约束以保证引用完整性。优先使用自增整数或 UUID。
3. **选择合适的数据类型（Data Types）** — 使用最小满足需求的类型（如 INT vs BIGINT），日期用 TIMESTAMP/DATE 而非字符串，JSON 数据谨慎使用 JSON 类型。
4. **索引策略（Indexing）** — 为 WHERE、JOIN、ORDER BY 列创建索引，但避免过度索引。定期分析查询计划（EXPLAIN）优化慢查询。
5. **考虑反范式化场景（Denormalization When Needed）** — 读密集型场景（如 BI、报表）可适度反范式化，以空间换时间。但核心事务数据应保持范式化。

## 标准代码示例

### 3NF 范式化表设计

```sql
-- 1NF: 消除重复组
-- 2NF: 消除部分依赖
-- 3NF: 消除传递依赖

CREATE TABLE customers (
    customer_id SERIAL PRIMARY KEY,
    customer_name VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE products (
    product_id SERIAL PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    supplier_id INT REFERENCES suppliers(supplier_id),
    price DECIMAL(10, 2) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE suppliers (
    supplier_id SERIAL PRIMARY KEY,
    supplier_name VARCHAR(100) NOT NULL,
    contact_email VARCHAR(100),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE orders (
    order_id SERIAL PRIMARY KEY,
    customer_id INT NOT NULL REFERENCES customers(customer_id),
    order_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    status VARCHAR(20) DEFAULT 'pending'
);

CREATE TABLE order_items (
    order_item_id SERIAL PRIMARY KEY,
    order_id INT NOT NULL REFERENCES orders(order_id) ON DELETE CASCADE,
    product_id INT NOT NULL REFERENCES products(product_id),
    quantity INT NOT NULL CHECK (quantity > 0),
    unit_price DECIMAL(10, 2) NOT NULL
);

-- 索引优化
CREATE INDEX idx_orders_customer ON orders(customer_id);
CREATE INDEX idx_orders_date ON orders(order_date);
CREATE INDEX idx_order_items_order ON order_items(order_id);
CREATE INDEX idx_products_supplier ON products(supplier_id);
```

### 带有软删除和审计字段的设计

```sql
CREATE TABLE users (
    user_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    
    -- 审计字段
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    created_by VARCHAR(50),
    updated_by VARCHAR(50),
    
    -- 软删除
    deleted_at TIMESTAMP,
    is_active BOOLEAN DEFAULT TRUE,
    
    -- 约束
    CONSTRAINT chk_email_format CHECK (email ~* '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$')
);

-- 自动更新 updated_at 触发器
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = CURRENT_TIMESTAMP;
    RETURN NEW;
END;
$$ language 'plpgsql';

CREATE TRIGGER update_users_updated_at 
    BEFORE UPDATE ON users 
    FOR EACH ROW 
    EXECUTE FUNCTION update_updated_at_column();
```

## 常见反模式

| 反模式 | 描述 | 后果 |
|--------|------|------|
| **EAV（Entity-Attribute-Value）反模式** | 将结构化数据存储为键值对（如 `attributes` 表）会丧失类型安全、约束验证和查询性能 | 应使用适当的列设计 |
| **过度使用 SELECT *** | 检索不需要的列增加 I/O 和内存消耗，且破坏索引覆盖扫描优化 | 应明确列出所需列 |
| **忽略外键约束（为了"性能"）** | 在应用层保证数据一致性不可靠且容易出错 | 外键约束对性能影响极小，但能防止数据孤儿和损坏 |
