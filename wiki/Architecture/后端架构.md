## 核心原则

1. **Clean Architecture/Hexagonal** — 领域逻辑位于架构中心，依赖关系始终向内指向。外层（框架、UI、数据库）依赖内层，反之禁止。
2. **依赖倒置（DIP）** — 高层模块不应依赖低层模块，两者都应依赖抽象。领域层定义接口（Ports），基础设施实现（Adapters）。
3. **领域优先（Domain First）** — 先设计领域模型，再考虑持久化、API等技术细节。如果能在没有数据库和UI的情况下运行领域逻辑，边界就是正确的。
4. **SOLID原则** — 单一职责、开闭原则、里氏替换、接口隔离、依赖倒置。这些原则在类级别和系统级别都适用。
5. **从简单开始，逐步演进** — 不要过度工程。从模块化单体开始，当团队规模和复杂度需要时再演进为微服务。

## 标准代码示例

### 领域实体（纯Java，无框架依赖）

```java
// domain/Account.java
public class Account {
    private AccountId id;
    private Money baselineBalance;
    private ActivityWindow activityWindow;
    
    // 业务行为封装在实体中
    public boolean withdraw(Money money, AccountId targetAccountId) {
        if (!mayWithdraw(money)) {
            return false;
        }
        
        Activity withdrawal = new Activity(
            this.id,
            this.id,
            targetAccountId,
            LocalDateTime.now(),
            money
        );
        this.activityWindow.addActivity(withdrawal);
        return true;
    }
    
    private boolean mayWithdraw(Money money) {
        return Money.add(
            this.calculateBalance(),
            money.negate()
        ).isPositiveOrZero();
    }
    
    public Money calculateBalance() {
        return Money.add(
            this.baselineBalance,
            this.activityWindow.calculateBalance(this.id)
        );
    }
}
```

### 应用服务（编排用例）

```java
// application/service/SendMoneyService.java
@RequiredArgsConstructor
public class SendMoneyService implements SendMoneyUseCase {
    
    private final LoadAccountPort loadAccountPort;
    private final UpdateAccountStatePort updateAccountStatePort;
    
    @Override
    public boolean sendMoney(SendMoneyCommand command) {
        // 校验
        requireNonNull(command);
        
        // 加载领域对象
        Account sourceAccount = loadAccountPort.loadAccount(
            command.getSourceAccountId(),
            LocalDateTime.now().minusDays(10)
        );
        Account targetAccount = loadAccountPort.loadAccount(
            command.getTargetAccountId(),
            LocalDateTime.now().minusDays(10)
        );
        
        // 执行业务操作
        boolean sourceResult = sourceAccount.withdraw(
            command.getMoney(), 
            targetAccount.getId()
        );
        if (!sourceResult) return false;
        
        boolean targetResult = targetAccount.deposit(
            command.getMoney(), 
            sourceAccount.getId()
        );
        if (!targetResult) return false;
        
        // 持久化
        updateAccountStatePort.updateActivities(sourceAccount);
        updateAccountStatePort.updateActivities(targetAccount);
        
        return true;
    }
}
```

## 常见反模式

| 反模式 | 描述 | 后果 |
|--------|------|------|
| **分布式单体（Distributed Monolith）** | 微服务共享数据库、同步调用链、必须一起部署 | 获得分布式的复杂度却没有获得自治性，调试和部署困难 |
| **过度工程（Over-engineering）** | 为几百笔日交易量的系统使用事件溯源+CQRS+微服务网格 | 两名开发者花费80%时间调试配置，系统难以维护 |
| **贫血领域模型** | 实体只有getter/setter，业务规则散落在Service中 | 领域逻辑分散、重复，难以测试和演进 |
