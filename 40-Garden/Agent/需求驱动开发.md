# 需求驱动开发

## 概述
一种使用 LLM 深度探索需求的实践技术：让 LLM 生成 100-500 个详尽问题组成的需求表，用户逐一回答后解耦为 PRD 和 Spec，再驱动开发。

## 核心思想

传统的需求描述只是大纲。LLM 可以帮助将大纲扩展为穷尽性的问题列表，从**开发角度**和**产品角度**两个维度深化需求。

### 好的需求应包含

- 名称、形态、边界
- 功能、适用人群
- 目标、性能、安全

## 提示词模板

```markdown
基于上述需求大纲，通过列表式对话，为我总结出 200-500 项详尽的问题表。
从开发角度和产品描述角度出发，对项目各方面进行完善。

注意：
1. 问题要足够细致、深度思考，提供足够多内容供参照
2. 参照市面上同类产品的优秀最佳实践进行补充
3. 对技术难点的具体解决方案，参照现有库/模块做集成化实现
4. 遵循冗余开发原则，不限制模型的泛化能力和边界
```

## 关键原则

1. **AI 不能替代人的需求定义**：从 0 到 1 的需求必须由人思考确定。AI 只能做现有方案的整合和优化
2. **答案确认环节不可省略**：问题表填完后，必须人机协同确认答案，不能容忍幻觉——这会影响整个开发方向
3. **两轮模型校验**：opus 4.7 生成 → gpt-5.5 补充，确保 spec 完善

## 工作流

```
需求大纲 → LLM 生成 100-500 问题表 → 人填写答案
    → 人机协同确认答案
    → LLM 解耦为 PRD + Spec
    → 多模型补充校验
    → 驱动开发
```
